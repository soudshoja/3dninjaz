---
title: Managing colours
category: Products
tags: [colours, library, variants, picker, bambu, polymaker]
order: 5
---

## What this is

A central, reusable colour library at **/admin/colours**. Colours are seeded once from `Colours/bambu-lab-colors.html` and `Colours/polymaker-colors.html` into the `colors` table, then surfaced anywhere the variant editor renders a Colour-named option.

The library is admin-only data: brand, family type/subtype, RFID/SKU codes, and old-vs-current hex are visible to you here. Customers see only **name** and **hex** — never codes, never family metadata.

## How it fits the variant system

A "Colour" option on a product behaves like Size or Material — it counts as one of the 6 variant axes and joins the cartesian generator on equal footing. The difference is the **picker**: when an option is named "Color" or "Colour" (case-insensitive), the variant editor adds a "Pick from library" button next to the freeform "Add value" input. That button opens a searchable picker drawing from the `is_active = true` colours.

Colours that come from the library snapshot the **name** and **hex** into the underlying `product_option_values` row at attach time. The library row is the source of truth for cascade renames (see below). The freeform path remains for one-off custom colours that aren't worth promoting to the library.

## Managing the library

### Add a colour
1. Click **+ New colour** at /admin/colours
2. Fill in: name (≤ 64 chars), hex (with the native colour picker), brand, family type/subtype, optional code (Bambu RFID or Polymaker SKU)
3. Optional: previous hex if the colour reformulated (Polymaker `oldHex`)
4. Active is on by default — uncheck to create as archived

### Edit a colour
- Open the row's 3-dot menu → **Edit**
- Saving cascades **name** and **hex** changes to every linked product variant value (see "Cascade rename" below). Other field changes (brand, family, code, previous hex, active flag) update the library row only.

### Archive (soft-delete) — always allowed
- 3-dot menu → **Archive**. Sets `is_active = false`. Archived colours disappear from the picker but stay on existing products (PDP and /shop continue to render them). Reactivate any time.

### Hard-delete with in-use guard
- 3-dot menu → **Delete…** → confirm
- If the colour is used by ≥1 product, the modal swaps to "Cannot delete — in use" and lists the affected products with quick links. The CTA changes to "Archive instead", which is one click.
- If no products reference the colour, deletion succeeds and the row is removed from the DB.

### Cascade rename — diff-aware
Renaming a library colour from "Galaxy Black" → "Cosmic Black" updates every `product_option_values` row that:
- Has `color_id` pointing at this library row, AND
- Still shows `value = "Galaxy Black"` (i.e. you haven't manually renamed it on a product)

If you renamed a specific product's value to "Black Hole" earlier, **that row is preserved** — your manual edit wins. Hex changes follow the same rule. The whole cascade runs in a single transaction; if it would touch >1000 variant rows you'll get a warning instead of an automatic write.

## Picker walk-through (per product)

1. On a product's variant editor, add an option named **Colour** (or **Color** — case-insensitive)
2. The "Pick from library" button appears. Click it.
3. The modal loads all active library colours. Search by name, code, or family. Filter by brand or family type.
4. Tick 1..N colours. The footer counter and primary button update — "Add 3 colours".
5. Click confirm. The picker closes; the editor refetches and the new values are now part of the option.

### Custom one-off (not in library)
Below the picker the existing freeform inputs remain, labelled **"Custom (not in library)"**. Use them for colours you don't want to promote into the library. Custom values store `color_id = NULL` and won't appear on /shop or in cross-product reuse — but they render correctly on the PDP.

### Already-attached guard
If a library colour is already on this option, its picker row appears greyed-out with the tooltip "Already attached to this product". You can't add the same library colour twice to one option.

## Customer view

### PDP swatch grid
On the PDP, customers see one swatch per available colour: a 32px hex circle with the colour name printed below — always visible, no hover required. Out-of-stock variants show a diagonal grey overlay and the caption goes line-through. The customer never sees the code, family, or previous hex — those stay admin-only.

### /shop sidebar Colour filter
Customers can filter /shop by colour. The Colour accordion lives below the Category list (default open, first 12 chips visible, Show all expands). Each chip is a 12px hex circle + name pill. Tapping a chip toggles the slug into the URL: `/shop?colour=galaxy-black` or `/shop?colour=galaxy-black,jade-white` for multi-select. Combine with category for an intersect: `/shop?category=keychains&colour=galaxy-black` returns only Galaxy Black keychains.

The chip list is computed live — only colours used by ≥1 active in-stock product appear. Archive a colour, or unattach it from every product, and its chip vanishes from /shop on the next render.

## Re-importing or re-seeding

The seed script `scripts/seed-colours.ts` is **idempotent**: re-running it after the first import is a full no-op (0 inserts, 0 updates) when the source HTML hasn't changed. If you edit the HTML to add a colour, re-running picks up the addition without disturbing existing rows. For one-off additions, prefer the admin form — the seed is designed for the initial ~145-row import only.

## Edge cases worth knowing

- **Em-dash `—` codes**: Polymaker's `Matte Rose (legacy)` has a literal em-dash placeholder. The seed treats em-dash as `null` code and uses `(brand, name)` as the natural key.
- **Polymaker dual & gradient sections**: skipped at seed time (multi-hex shape doesn't fit the single hex column). Add manually if needed.
- **Cross-brand name collisions**: if both Bambu and Polymaker have a colour named "Black", the /shop URL slugs become `black-bambu` and `black-polymaker` to keep them distinct. Single-brand colours just use the simple slug `black`.

## Where the data lives

- `colors` (library): the central table you manage at /admin/colours
- `product_option_values.color_id`: nullable FK back to `colors.id`. Non-null = library colour (cascade-aware). Null = custom freeform.
- `product_option_values.value` and `swatch_hex`: snapshots — the cascade keeps them in sync with the library when you rename. Manual edits to these per-product win over the cascade.
