Create a reusable server-safe link component (it renders a plain anchor — no client state):

```tsx
import Link from "next/link";
import { whatsappLink, BUSINESS } from "@/lib/business-info";

type Props = {
  message?: string;
  variant?: "primary" | "ghost" | "inline";
  className?: string;
  children?: React.ReactNode;
};

export function WhatsAppCta({ message, variant = "primary", className = "", children }: Props) {
  const href = whatsappLink(message);
  // Anchor: target=_blank, rel=noopener noreferrer
  // Primary variant: chunky green pill, 48px+ tap height
  // Include WhatsApp SVG icon inline (no external dependency) beside the label for primary/ghost variants
  return (
    <Link
      href={href}
      target="_blank"
      rel="noopener noreferrer"
      aria-label={`Chat with 3D Ninjaz on WhatsApp at ${BUSINESS.whatsappNumberDisplay}`}
      className={/* tailwind per variant */}
    >
      {children ?? "Chat on WhatsApp"}
    </Link>
  );
}
```

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Static content → public browser | Privacy/Terms/About/Contact pages are public, indexable; any leaked PII in copy is a disclosure problem |
| wa.me deep link | External redirect; user leaves our site; number and pre-filled message are reflected in the URL |
| Register form → /privacy link | Consent flow's integrity depends on /privacy loading real content, not 404 |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-02-01 | I (Information Disclosure) | Privacy/Terms pages | mitigate | Pages render ONLY hard-coded JSX + whitelisted BUSINESS constant fields. No user-supplied content, no Markdown rendering, no admin-controlled copy. Prevents stored-XSS and accidental PII leakage via CMS-style input. |
| T-04-02-02 | T (Tampering) | WhatsApp deep link | mitigate | `whatsappLink(message)` uses `encodeURIComponent` on the message argument. Message parameter is always a hard-coded string from page components — no user input flows in. Even if future code passes user input, encoding neutralises injection. |
| T-04-02-03 | S (Spoofing) | wa.me number placeholder | mitigate | Plan 04-04 launch checklist treats `60000000000` as a blocker — launch cannot proceed with a placeholder number that would point customers to the wrong WhatsApp account. Task 1 adds a TODO comment on the placeholder line flagging it as a blocker. |
| T-04-02-04 | I (Information Disclosure) | WhatsApp link on public pages | accept | Publishing a WhatsApp number publicly is the business decision; this is standard for MY SMEs. Mitigation: Privacy Policy notes that contacting us via WhatsApp is voluntary and subject to WhatsApp's own privacy policy. Social-engineering risk (attacker spoofs our number) mitigated by also publishing the same number in About + Contact + Terms — customers can cross-reference. |
| T-04-02-05 | R (Repudiation) | PDPA consent record | mitigate | Phase 1 already captures `pdpaConsentAt` timestamp at registration. Privacy policy's "Consent record" section explains this so users are aware. No change needed here — documented. |
| T-04-02-06 | I (Information Disclosure) | PDPA policy completeness | mitigate | Policy MUST cover: data collection notice, consent reference, purpose limitation, retention, deletion/correction rights, third parties, DPO contact, 21-day response. Task 3 explicitly lists all 11 sections. Verify task checklist enforces presence. |
| T-04-02-07 | T (Tampering) | Static content routes | accept | `/about`, `/contact`, `/privacy`, `/terms` are static server components with no database reads and no user input paths. Tampering requires repo access, which is not part of the web threat surface. |
| T-04-02-08 | E (Elevation of Privilege) | target="_blank" on WhatsApp link | mitigate | All external links use `rel="noopener noreferrer"` to prevent reverse tabnabbing and referrer leakage. Enforced in Task 1 verify step. |
