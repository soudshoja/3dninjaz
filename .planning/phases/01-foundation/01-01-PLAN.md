3. Create `src/lib/db/schema.ts` with ALL tables. Better Auth requires specific tables (user, session, account, verification). IMPORTANT: Use `@better-auth/drizzle` to generate the auth schema if available, OR manually define per Better Auth Drizzle adapter docs. Then add application tables alongside.

The schema MUST include these Better Auth tables (per https://better-auth.com/docs/adapters/drizzle):
- `user` table: id (text PK), name, email (unique), emailVerified, image, createdAt, updatedAt, role (text, default "customer" per D-08), pdpaConsentAt (timestamp, nullable, per D-09/AUTH-05)
- `session` table: id (text PK), expiresAt, token (unique), ipAddress, userAgent, userId (FK to user)
- `account` table: id (text PK), accountId, providerId, userId (FK to user), accessToken, refreshToken, idToken, accessTokenExpiresAt, refreshTokenExpiresAt, scope, password, createdAt, updatedAt
- `verification` table: id (text PK), identifier, value, expiresAt, createdAt, updatedAt

Application tables (per D-10, D-11, D-12, D-13):
- `categories` table: id (uuid PK, defaultRandom), name (text, unique, not null), slug (text, unique, not null), createdAt (timestamp, defaultNow)
- `products` table: id (uuid PK, defaultRandom), name (text, not null), slug (text, unique, not null), description (text, not null), images (text array, default empty, max 5 enforced at app level per D-03), materialType (text, nullable), estimatedProductionDays (integer, nullable), isActive (boolean, default true, per ADM-04), isFeatured (boolean, default false, per D-12), categoryId (uuid FK to categories, nullable), createdAt (timestamp, defaultNow), updatedAt (timestamp, defaultNow)
- `productVariants` table (per D-13): id (uuid PK, defaultRandom), productId (uuid FK to products, onDelete cascade, not null), size (text enum ["S","M","L"], not null), price (decimal precision 10 scale 2, not null -- MYR), widthCm (decimal precision 6 scale 1, nullable), heightCm (decimal precision 6 scale 1, nullable), depthCm (decimal precision 6 scale 1, nullable)

Also define Drizzle `relations()` for all tables:
- products -> category (one), products -> variants (many)
- productVariants -> product (one)
- categories -> products (many)

CRITICAL: The Better Auth tables must use the EXACT column names and types that Better Auth expects. Check the Better Auth Drizzle adapter docs carefully. If Better Auth provides a CLI to generate schema (`npx @better-auth/cli generate`), run it first and then manually add the `role` and `pdpaConsentAt` fields plus the application tables.

4. Create `src/lib/auth.ts` -- Better Auth server configuration per Research Pattern 1:
```typescript
import { betterAuth } from "better-auth";
import { drizzleAdapter } from "better-auth/adapters/drizzle";
import { admin } from "better-auth/plugins";
import { db } from "@/lib/db";
import * as schema from "@/lib/db/schema";

export const auth = betterAuth({
  database: drizzleAdapter(db, {
    provider: "pg",
    schema: schema,
  }),
  emailAndPassword: {
    enabled: true,
    sendResetPassword: async ({ user, url }) => {
      // Use Resend for password reset emails (AUTH-04)
      await fetch("https://api.resend.com/emails", {
        method: "POST",
        headers: {
          Authorization: `Bearer ${process.env.RESEND_API_KEY}`,
          "Content-Type": "application/json",
        },
        body: JSON.stringify({
          from: "Print Ninjaz <noreply@printninjaz.com>",
          to: user.email,
          subject: "Reset your Print Ninjaz password",
          html: `<p>Hi ${user.name},</p><p>Click <a href="${url}">here</a> to reset your password. This link expires in 1 hour.</p><p>If you did not request this, ignore this email.</p>`,
        }),
      });
    },
  },
  user: {
    additionalFields: {
      pdpaConsentAt: {
        type: "date",
        required: false,
        input: true,
      },
    },
  },
  plugins: [
    admin({
      defaultRole: "customer",
    }),
  ],
});
```

NOTE: If Better Auth admin plugin does not accept "customer" as defaultRole (Research Open Question 3), fall back to using "user" as the default role. Test this during execution -- if `admin({ defaultRole: "customer" })` throws an error, use `admin()` with default settings (which defaults to "user" role) and treat "user" semantically as "customer".

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Client -> Auth API | Untrusted signup/signin requests cross to Better Auth handler |
| Auth -> Database | Session/user data written to Neon PostgreSQL |
| Admin seed script | Elevated privilege operation creating admin user |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-01 | Spoofing | Auth API route | mitigate | Better Auth handles password hashing, session tokens, and cookie security. HTTP-only cookies prevent XSS token theft. |
| T-01-02 | Tampering | .env.local | mitigate | .env.local in .gitignore, never committed. Placeholder values in template only. |
| T-01-03 | Information Disclosure | DATABASE_URL | mitigate | Connection string only in .env.local (gitignored). Neon enforces SSL (sslmode=require). |
| T-01-04 | Elevation of Privilege | User role field | mitigate | Role defaults to "customer" in both Better Auth config and DB schema. Admin only assignable via seed script, not registration API. |
| T-01-05 | Repudiation | PDPA consent | mitigate | pdpaConsentAt stored as server-side timestamp, not client boolean. Cannot be forged or backdated. |
</threat_model>
