<objective>
Extend the **cart store + bag UI** to carry per-line `configurationData` for made-to-order products. Behavior contract:
- **Stocked product addItem path:** UNCHANGED. variantId-based dedupe untouched (per D-14 — ≤5 additive lines in the variant code path of cart-store.ts).
- **Configurable product addItem path:** identical configurationData JSON (stable-stringified + sha-1-hashed in `src/lib/config-hash.ts`) → qty++ on existing line. Different hash → new line. Each line key encodes `${variantId|productId}::${configHash}` so the React list-key + dedupe stay consistent.
- **Cart drawer + /bag row UI:** renders `configurationData.computedSummary` underneath the product name (e.g., `"\"JACOB\" (5 letters) · Red base+chain · White letters"`), and uses `configurationData.computedPrice` as `unitPrice` for the line.
- **Server hydration (`hydrateCartItems` in `src/actions/cart.ts`):** for configurable lines, reads `configurationData` from the persisted cart payload (NOT from DB — DB stocks no per-line config until checkout) and surfaces it in the `HydratedCartItem` for rendering. Stocked lines unchanged.

Per CONTEXT.md D-11 the cart-line `variantId` field stays mandatory for stocked products. For configurable products there is **no variant** — the cart line carries `productId` + `configurationData` instead. Use a discriminated union (see interfaces below).

Per D-14 BLOCKING: variant-flow code path adds ≤5 lines. The new fields (`productId`, `configurationData`) are additive type-widenings; the existing `addItem({ variantId })` shape continues to compile and behave as before.

Purpose: REQ-8 + REQ-9. Without this plan, customers cannot add a configured keychain to bag, and qty-bump-vs-new-line dedupe doesn't work. Plan 19-09 reads the same `configurationData` at PayPal capture to snapshot into `order_items`.
Output: 1 cart-store extension + 1 hash helper + 1 modified hydrateCartItems server action + cart-drawer.tsx + /bag page row rendering of computedSummary.
</objective>
