<objective>
Capstone plan: ship the **Custom Name Keychain seed product** + admin-guide article + 24-step manual smoke checklist for the verifier. This plan has zero code modifications — it produces the **proof artifacts** that close every Phase 19 requirement (REQ-1 through REQ-9).

Three deliverables:

1. **`scripts/seed-keychain-product.ts`** — idempotent (slug check `custom-name-keychain`); inserts: 1 product (productType=configurable, maxUnitCount=8, priceTiers `{1:7,2:9,3:12,4:15,5:18,6:22,7:26,8:30}`, unitField="name", 1 primary display image at `public/uploads/products/seed-keychain/orig.png` if it exists else placeholder generation), 1 text config field "Your name" (max=8, allowedChars="A-Z", uppercase=true, profanityCheck=true), 2 colour config fields (Base+chain — 5 allowedColorIds; Letter — 3 allowedColorIds). Reads colour ids by name from the colours table seeded by Phase 18.

2. **`scripts/seed-profanity.ts`** — idempotent seed for the profanity allowlist (D-13). Stores 25-30 starter words in the chosen storage location (whichever Plan 19-02 / Plan 19-04 chose: site_settings JSON column OR a dedicated `profanity_words` table — confirm by reading those plans before writing this script).

3. **`src/content/admin-guide/products/made-to-order.md`** — admin guide article covering: when to use made-to-order vs stocked, how to build the configurator, how to set tier prices, how to test the customer flow. Mirrors the existing `src/content/admin-guide/products/colours.md` structure (Plan 18-09 pattern).

4. **`.planning/phases/19-made-to-order-product-type/19-SMOKE-CHECKLIST.md`** — 24 hand-numbered manual steps the verifier walks through to prove every acceptance criterion in 19-SPEC.md. Mirrors the Phase 18 verifier-handoff format.

Per CONTEXT.md D-15: seed is idempotent (slug-based dedup). Per D-13: profanity allowlist seeded with conservative word list; admin can edit later via a future settings page (out of scope for v1).

Purpose: the keychain is the smoke fixture for every Phase 19 acceptance check — without it, plan 19-09 / 19-10 verification has no real-world data to validate against. The admin guide closes the operational documentation gap. The smoke checklist is the verifier's runbook.
Output: 2 idempotent seed scripts + 1 admin guide article + 1 verifier checklist.
</objective>
