<objective>
Persist `configurationData` through the order pipeline at PayPal capture time, then render the resulting summary in **all four order-rendering surfaces**: admin order detail, customer order detail, invoice PDF, and order-confirmation email (HTML + text).

Per Plan 19-01: `order_items.configurationData` LONGTEXT NULL column already exists. This plan wires:
1. **Capture-time write:** `capturePayPalOrder` (in `src/actions/paypal.ts`) reads `configurationData` from each cart line via `hydrateCartItems` (which Plan 19-08 already extended). When a cart line has `configurationData`, the inserted `order_items` row stores `JSON.stringify(configurationData)` in the new column. Stocked lines write NULL — UNCHANGED.
2. **Read-time helper:** `ensureOrderItemConfigData(raw)` parses LONGTEXT → `ConfigurationData | null` (mirrors `ensureImagesArray` / `ensureTiers` / `ensureConfigJson` family — manual JSON parse with shape validation). Lives in `src/lib/config-fields.ts` (extension of Plan 19-02).
3. **Four render surfaces:** wherever the existing code renders `i.variantLabel ?? (i.size ? "Size " + i.size : null)`, add a precedence rule: **if `configurationData?.computedSummary` exists, use it INSTEAD of `variantLabel`**. Reason: the summary is more informative ("\"JACOB\" (5 letters) · Red base+chain · White letters") than `variantLabel` (which for configurable lines is set to `computedSummary` in cart-drawer rendering already, but `order_items.variantLabel` will still hold the literal `computedSummary` string snapshotted at capture time — so there's no actual code-level branch needed for cart→order). However, **for clarity + consumer-facing labels**, the order-rendering surfaces should use the parsed `configurationData.computedSummary` directly to avoid relying on the variantLabel column being populated.

Per CONTEXT.md D-12: stocked rows continue to write NULL into `configurationData` column — no behavior change. Per D-14: variant code path receives ZERO additional lines in this plan (the only changed file at the variant boundary is `src/actions/paypal.ts`, where the new lines are wholly inside the configurable branch).

Purpose: REQ-8. Without this plan the configuration is dropped at checkout — admin printer never sees what to print. This is the "customer pays MYR 18 → admin gets the manifest" plumbing.
Output: 1 capture-time write + 1 parse helper extension + 4 read-time render branches.
</objective>

       ```typescript
       export function ensureOrderItemConfigData(raw: unknown): ConfigurationData | null {
         if (raw == null || raw === "") return null;
         let parsed: unknown;
         if (typeof raw === "string") {
           try { parsed = JSON.parse(raw); } catch { return null; }
         } else {
           parsed = raw;
         }
         if (!parsed || typeof parsed !== "object") return null;
         const o = parsed as Record<string, unknown>;
         if (typeof o.computedPrice !== "number") return null;
         if (typeof o.computedSummary !== "string") return null;
         if (!o.values || typeof o.values !== "object") return null;
         // Shallow value check — every value must be a string
         for (const v of Object.values(o.values as Record<string, unknown>)) {
           if (typeof v !== "string") return null;
         }
         return {
           values: o.values as Record<string, string>,
           computedPrice: o.computedPrice,
           computedSummary: o.computedSummary,
         };
       }
       ```
    4. Run tests — must pass.
    5. Add Zod-equivalent shape check or keep manual (manual is simpler + bundle-cheap; mirrors existing `ensureXxx` family style).
