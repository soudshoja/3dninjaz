# Plan 16-03 — Admin Variant Editor UI (Wave 2)

**Depends on:** 16-02
**Parallel with:** 16-04

## Goal

New route `/admin/products/[id]/variants` with: (1) options editor (add/rename/delete option + values), (2) variant matrix generator (auto-expand from options cartesian), (3) per-variant inline editor (price, stock, SKU, image upload, inStock toggle).

## Tasks

1. **Server actions — `src/actions/variants.ts`**
   - `addProductOption(productId, name)` — inserts option at next position; errors if position>3
   - `renameProductOption(optionId, name)` — Zod + admin auth
   - `deleteProductOption(optionId)` — cascade drops values + variants using this option slot (warn in UI first)
   - `addOptionValue(optionId, value, swatchHex?)` — insert at next position
   - `renameOptionValue(valueId, value, swatchHex?)`
   - `deleteOptionValue(valueId)` — must first soft-delete variants using this value; UI surfaces "3 variants will be deleted, confirm?"
   - `generateVariantMatrix(productId)` — reads options + values, computes cartesian; for each missing combo inserts a `productVariants` row with `price=0`, `stock=0`, `inStock=false`, `label_cache` composed; returns count inserted
   - `updateVariant(variantId, fields)` — Zod-validated partial update: price, stock, inStock, sku, imageUrl, position, costPrice + breakdown fields (Phase 14 passthrough)
   - `deleteVariant(variantId)` — hard delete (orders snapshot label already)
   - `uploadVariantImage(variantId, file)` — uses existing image pipeline (`src/lib/image-writer.ts` per Phase 7); stores `image_url`
   - All actions gated by `requireAdmin()` as first `await` (CVE-2025-29927 precedent)

2. **Page `src/app/admin/products/[id]/variants/page.tsx`** (server component)
   - Fetches product + hydrated options/variants via `hydrateProductVariants`
   - Renders `<VariantEditor>` client component with initial data

3. **Client component `src/components/admin/variant-editor.tsx`**
   - Top section: options editor
     - Lists current options (max 3)
     - "Add Option" button (disabled at 3)
     - Each option card: rename inline, value list with add/rename/delete, drag-reorder values (optional v1)
     - Color option: swatch picker for each value (hex input + preview)
   - Middle: "Generate variant matrix" button (disabled if any option has 0 values)
   - Bottom: variant table
     - Columns: Option combo label | Price | Stock | Track Stock | SKU | Image | Active | Actions
     - Inline edit with debounced save
     - "Regenerate matrix" button — adds missing combos, doesn't touch existing
     - "Delete" per row
   - Uses existing shadcn/ui components (Table, Dialog, Form, Input, Button, Switch)
   - Uses existing Zustand/TanStack Query pattern per project conventions
   - Toast on each save via existing toaster

4. **Entry link in product form**
   - `src/app/admin/products/[id]/page.tsx` (or form component): add "Manage Variants →" link after initial save; disabled during product creation (needs product ID)

5. **Remove legacy size-price UI from main product form**
   - Keep in place BUT hidden behind a feature gate OR replaced with a "Migrated to new variant model" notice linking to the new variants page
   - Old S/M/L price inputs still write via the dual-read path during rollout

- [x] Admin navigates to `/admin/products/[id]/variants` and sees option editor + current variants
- [x] Admin adds a new option ("Color") with 3 values (Red, Blue, Green) and clicks "Generate Matrix" — system creates 3 × 3 = 9 variants (or N × M matching current options)
- [x] Admin edits price of one variant inline; refresh confirms persistence
- [x] Admin uploads an image for one variant; image renders in variant row
- [x] Admin deletes one option value; UI warns count + deletes affected variants on confirm
