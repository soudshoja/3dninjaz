# Plan 17-01 — Schema + Zod + Raw-SQL Migration (Wave 1)

**Depends on:** Phase 16 complete
**Unblocks:** 17-02

## Goal

Land five new columns on `product_variants` (`sale_price`, `sale_from`, `sale_to`, `is_default`, **`weight_g`** per AD-08) via a raw-SQL migration applicator that mirrors the Phase 16 pattern. Update `src/lib/db/schema.ts` and `src/lib/variants.ts` types. Extend `variantUpdateSchema` in Zod. No code paths change yet — this plan is purely data-model groundwork for 17-02.

## Tasks

### 1. Schema edits in `src/lib/db/schema.ts`

Extend `productVariants` table definition. Insert new columns AFTER the existing Phase 16 block (positional — match the migration script's `AFTER` clauses):

```ts
// Phase 17 — sale pricing + default-variant flag + per-variant shipping weight
salePrice: decimal("sale_price", { precision: 10, scale: 2 }),
saleFrom: timestamp("sale_from"),
saleTo: timestamp("sale_to"),
isDefault: boolean("is_default").notNull().default(false),
// AD-08 — per-variant shipping weight in grams. NULL = inherit product.shippingWeightKg × 1000.
weightG: int("weight_g"),
```

Comment block above explaining:
- `salePrice`: optional lower price. Effective price = `salePrice ?? price` when the sale window is active.
- `saleFrom`, `saleTo`: UTC TIMESTAMPs. Each nullable — NULL means "no bound on that side". Both NULL = active as soon as `salePrice` set.
- `isDefault`: admin-marked default combo; at most one per product (app-layer transaction enforced in `setDefaultVariant`).
- `weightG` (AD-08): per-variant Delyva shipping weight override in grams. NULL means inherit `products.shippingWeightKg × 1000`; if product weight is also NULL, `quoteForCart` falls back to 500 g + emits a `warn` log.

### 2. Update `HydratedVariant` type in `src/lib/variants.ts`

Add fields:

```ts
salePrice: string | null;
saleFrom: Date | null;
saleTo: Date | null;
isDefault: boolean;
/** Derived at hydration time: effective_price given server-now. */
effectivePrice: string;
/** Derived: true if sale_price set AND now inside [sale_from, sale_to]. */
isOnSale: boolean;
```

In `hydrateProductVariants`, compute `effectivePrice` and `isOnSale` inline:

```ts
const now = new Date();
const isOnSale =
  v.salePrice !== null
  && (v.saleFrom === null || v.saleFrom.getTime() <= now.getTime())
  && (v.saleTo   === null || v.saleTo.getTime()   >= now.getTime());
const effectivePrice = isOnSale ? (v.salePrice as string) : v.price;
```

Apply in BOTH the zero-options fallback branch AND the hydrated branch (two emit sites).

### 3. Zod update in `src/lib/validators.ts`

Extend `variantUpdateSchema` (line 639) with:

```ts
salePrice: z
  .string()
  .regex(/^\d+(\.\d{1,2})?$/, "Sale price must be a valid decimal")
  .optional()
  .or(z.literal(""))
  .nullable(),
saleFrom: z
  .string()
  .datetime({ offset: true })
  .optional()
  .or(z.literal(""))
  .nullable(),
saleTo: z
  .string()
  .datetime({ offset: true })
  .optional()
  .or(z.literal(""))
  .nullable(),
isDefault: z.boolean().optional(),
// AD-08 — optional per-variant shipping weight (grams). NULL/omitted = inherit product weight.
weightG: z.number().int().min(0).max(50000).nullable().optional(),
```

Add a `.refine` at schema level to reject `salePrice >= price` when both provided — BUT this requires `price` to be in the object; since `variantUpdateSchema` is a partial patch, defer price-vs-sale validation to the server action (`updateVariant` and `bulkUpdateVariants`). Document this in a JSDoc comment above `salePrice`.

DDL to apply, in order, via `addColumnIfMissing`:
