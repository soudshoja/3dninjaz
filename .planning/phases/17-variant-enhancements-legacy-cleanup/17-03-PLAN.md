# Plan 17-03 — PDP Sale Rendering + OOS Hardening + Default Pre-select + Image Swap (Wave 2)

**Depends on:** 17-01 (HydratedVariant has `effectivePrice`, `isOnSale`, `isDefault`)
**Unblocks:** 17-04 (legacy PDP branches removed only after new path proven)

## Goal

Ship customer-visible Phase 17 features on the product detail page:

1. Strikethrough regular price + bold sale price + "ON SALE" badge when `isOnSale`
2. Default variant pre-selected on PDP load when `is_default=true` exists
3. OOS values hardened — disabled, `title` tooltip, keyboard-unreachable
4. Gallery primary image swaps to selected variant's `imageUrl` when set; reverts to product images when variant cleared or has no image

Legacy PDP fallback (SizeSelector / SizeGuide / AddToBagButtonV2 wrapper) stays in place during this plan — removed in 17-04.

### Out of scope for this plan

- Per-variant weight (AD-08) is a server-side quote input only. PDP does NOT display weight. No PDP changes required for AD-08.

### 3. Default pre-selection in `src/components/store/variant-selector.tsx`

Update `defaultSelected` memo (line ~51):

```tsx
const defaultSelected = useMemo<SelectedValues>(() => {
  // 1. Explicit default
  const def = variants.find((v) => v.isDefault);
  if (def) return [...def.optionValueIds] as SelectedValues;
  // 2. First available
  const available = variants.find(isVariantAvailable);
  if (available) return [...available.optionValueIds] as SelectedValues;
  // 3. First variant
  const first = variants[0] ?? null;
  if (!first) return [null, null, null];
  return [...first.optionValueIds] as SelectedValues;
}, [variants]);
```

Rationale: admin-marked default wins over first-available (matches AD-05 intent).

### 5. Gallery image swap

In `src/components/store/product-detail.tsx`, the existing code (lines ~102–106):

```tsx
const variantImageUrl =
  hasGenericOptions && selectedHydrated?.imageUrl ? selectedHydrated.imageUrl : null;
const galleryImages = variantImageUrl
  ? [variantImageUrl, ...product.images.filter((i) => i !== variantImageUrl)]
  : product.images;
```

is close but treats the variant URL as a flat string. Phase 7 image pipeline expects a `baseUrl`; `<ProductGallery>` needs a `pictures` array when the variant has its own image.

Fix:

1. When `variantImageUrl` is set, call `pickImage(variantImageUrl)` to resolve a `PictureData` on the CLIENT side — but `pickImage` is server-only (reads `manifest.json` via fs).

2. Server resolution pre-hydration: in `src/app/(store)/products/[slug]/page.tsx`, after `hydrateProductVariants`, resolve a `PictureData` for every variant with `imageUrl`:

```tsx
const variantPictures = await Promise.all(
  hydratedVariants.map(async (v) =>
    v.imageUrl ? await pickImage(v.imageUrl) : null,
  ),
);
// pass down to ProductDetail as prop keyed by variantId
const variantPictureByVariantId = Object.fromEntries(
  hydratedVariants.map((v, i) => [v.id, variantPictures[i]]),
);
```

3. `ProductDetail` accepts new prop `variantPictures: Record<string, PictureData | null>`.

4. `galleryImages` logic becomes:

```tsx
const variantPic = selectedHydrated ? variantPictures[selectedHydrated.id] : null;
const effectivePictures = variantPic ? [variantPic, ...pictures.filter((p) => p.fallbackSrc !== variantPic.fallbackSrc)] : pictures;
```

5. `<ProductGallery pictures={effectivePictures} />` — it now receives a different array when selection changes; it re-renders with the variant image first.

**Alternative simpler path:** if `pickImage` dual-nature (file vs baseUrl) is an issue, fall back to the plain `imageUrl` as the fallbackSrc string inserted into the `images` string array — looses srcset but works. Preferred path: use `pickImage` server-side as above.

### 7. Cart hydration — respect effective price

`src/actions/cart.ts::hydrateCartItems` (from Phase 16-05) resolves `unitPrice` from the DB. Extend:

```ts
// When loading the variant row for a cart line, compute effectivePrice exactly
// as hydrateProductVariants does (sale window check on now = Date()).
const now = new Date();
const isOnSale =
  v.salePrice !== null
  && (v.saleFrom === null || new Date(v.saleFrom) <= now)
  && (v.saleTo   === null || new Date(v.saleTo)   >= now);
const unitPrice = isOnSale ? v.salePrice : v.price;
```

This ensures the customer is charged the effective price at each cart render AND at checkout capture. T-17-05-sale-in-flight is acknowledged (sale can end mid-purchase; customer is charged whatever is active at capture).
