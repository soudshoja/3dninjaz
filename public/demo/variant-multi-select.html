Your doubt — does multi-select variants alone fix the keycap product?

Honest caveman answer: mostly yes. You don't need a separate personalization layer for v1.

  • ✓ Solves: "I want 5 letters of JACOB" → multi-select letter axis → cart auto-creates 5 lines.
  • ✓ Solves: "I want each letter in 2 colour ways" → multi-select on colour axis too → cart = 5×2 = 10 lines.
  • ✓ Solves: stock control per letter / colour — each variant has its own stock count.
  • ✓ Solves: price per letter — each variant priced independently.
  • ✗ Limit: letter order. JACOB ≠ JBCAO. Customer ticks letters but not order. Fix: cart line carries orderIndex (1..N) auto-generated as customer ticks. Or one tiny text input "name to print: ___" feeding the multi-select.
  • ✗ Limit: if word repeats letters (BOBBY = B,O,B,B,Y), variant qty must be 3 for B. Fix: tick-counter on the Letter chip (tap once = 1, twice = 2, etc) OR a small text field that auto-increments tick counts.
  • ✗ Limit: if you want per-letter different colour (J=red, A=blue) → that's true configurator territory, not variants. v1 model is "all letters same colour set". 99% of customers want this anyway.

Recommended path:

  1. Build the pivot grid + multi-select admin (above).
  2. Add per-axis multi-select toggle on each option ("customer can pick multiple"). Default off.
  3. For keycap product, mark Letter axis as multi-select.
  4. Add a tiny text input "Type your name" that just helps tick the right letters in the right count. Feeds the variant multi-select. No new schema.

Defer the personalization layer until you hit a product that genuinely needs free-text input (engraving, custom logo). The keycap product does not.
